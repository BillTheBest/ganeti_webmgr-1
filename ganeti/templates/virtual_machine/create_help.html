The owner indicates who this virtual machine belongs to. Resources used by the virtual machine will be deducted from the owner's quota.

An owner may be a user or a group. If you are a member of a group you may create a virtual machine on behalf of that group.

If your user does not directly have access/permission to a cluster, it will not have permission to own a virtual machine directly.

Which ganeti cluster to deploy the new virtual machine on.

Fully qualified domain name (FQDN) to assign to this virtual machine. (e.g. hostname.example.org)

Uncheck this if you don't want the instance to automatically start after creation. If you do so, you can start it manually on the virtual machine detail page.

Check the virtual machine DNS name via the resolver (e.g. in DNS or /etc/hosts, depending on your setup). Since the name check is used to compute the IP address this also enables/disables IP checks (e.g. if the IP is pingable).

This is useful for setups that deploy virtual machines using dynamic DNS and thus the name is not resolvable yet.

Use with caution! If left unchecked you may run into name/ip collisions.

Automatically select primary and secondary node to allocate disks on.

When selected it will use the cluster default iallocator (if set by the cluster). The iallocator being used will be displayed after the checkbox.

Disk layout template for the virtual machine on the cluster node. The possible choices are:

  • plain - Disk devices will be logical volumes (e.g. LVM)
  • drbd - Disk devices will be DRBD (version 8.x) on top of LVM volumes
  • file - Disk devices will be regular files (e.g. qcow2)
  • diskless - This creates a virtual machine with no disks. Its useful for testing only (or other special cases).

If drbd is selected, then a primary and secondary node will need to be chosen unless automatic allocation has been selection. DRBD will allow the virtual machine to use live migration and failover in case one of the nodes goes offline.

The primary node to use for the virtual machine (in case automatic allocation is not used).

The secondary node to use for the virtual machine (in case automatic allocation is not used).

This is only required when using the drbd disk template.

Operating system to install on the virtual machine. Your choices are limited to the images configured on the cluster.

The text in bold signifies the Ganeti Operating System Type which may be called debootstrap, image, or some other type. The text that is selectable is the operating system (or os-type variant) that the cluster has access to.

This option changes the name of the Virtual Machine's Operating System. It does not reinstall the Operating System.

This option specifies how the virtual machine connects to the network. More information on this can be found in the Ganeti tutorial documentation.

When in doubt, choose bridged.

  • bridged - The virtual machine's network interface will be attached to a software bridge running on the node.
  • routed - The virtual machine's network interface will be routed

In bridged mode, it specifies the bridge interface to attach this NIC to on the node (e.g. br0).

In routed mode it's intended to differentiate between different routing tables/virtual machine groups (but the meaning is dependant on the network script, see gnt-cluster(8) for more details.

This parameter determines the way the network cards are presented to the virtual machine. The possible options are:

  • rtl8139 - (default for Xen HVM) (HVM & KVM)
  • ne2k_isa - (HVM & KVM)
  • ne2k_pci - (HVM & KVM)
  • i82551 - (KVM)
  • i82557b - (KVM)
  • i82559er - (KVM)
  • pcnet - (KVM)
  • e1000 - (KVM)
  • paravirtual - (default for KVM) (KVM & HVM)

Valid for the Xen PVM and KVM hypervisors.

This option specifies a MAC address to be associated with the NIC.

Acceptable values are a valid MAC address or 'generate' to generate a new unique MAC address.

A boolean option that specifies if the hypervisor should enable ACPI support for this instance. By default, ACPI is disabled.

Valid for the Xen HVM and KVM hypervisors.

The disk cache mode. Note that there are special considerations for the cache mode depending on version of KVM used and disk type (always raw file under Ganeti), please refer to the KVM documentation for more details. The possible options are: The possible options are:

  • None - For direct I/O
  • Default - No cache option is sent to KVM.
  • Writethrough - To use the host cache but report completion to the guest only when the host has committed the changes to disk
  • Writeback - To use the host cache and report completion as soon as the data is in the host cache

Valid for the KVM hypervisor.

This option specifies the path (on the node) to the initrd to boot the instance with. Xen PVM instances can use this always, while for KVM if this option is only used if the Kernel Path option is also specified.

You can pass here either an absolute filename (the path to the initrd) if you want to use an initrd, or use the format no_initrd_path for no initrd.

Valid for the Xen PVM and KVM hypervisors.

This options specifies extra arguments to the kernel that will be loaded. device. This is always used for Xen PVM, while for KVM it is only used if the kernel_path option is also specified.

The default setting for this value is simply "ro", which mounts the root disk (initially) in read-only one. For example, setting this to single will cause the instance to start in single-user mode.

Valid for the Xen PVM and KVM hypervisors.

This option passes the -mem-path argument to kvm with the path (on the node) to the mount point of the hugetlbfs file system, along with the -mem-prealloc argument too.

Valid for the KVM hypervisor.

The maximum amount of time (in ms) a KVM instance is allowed to be frozen during a live migration, in order to copy dirty memory pages.

Default value is 30ms, but you may need to increase this value for busy instances.

Note : This option is only effective with KVM versions >= 87 and QEMU-KVM versions >= 0.11.0.

Valid for the KVM hypervisor.

The security model for kvm. The possible options are:

  • None - The default option. Nothing is done and instances are run as the Ganeti daemon user (normally root).
  • User - KVM will drop privileges and become the user specified by the Security Domain parameter.
  • Pool - A global cluster pool of users will be used, making sure no two instances share the same user on the same node. (Note : This mode is not implemented yet).

Valid for the KVM hypervisor.

Specifies the username to run the instance under. It must be a valid username existing on the host.

Valid for the KVM hypervisor.

This option specifies the usb mouse type to be used. It can be "mouse" or "tablet". When using VNC it's recommended to set it to "tablet".

Valid for the KVM hypervisor.

This boolean option determines wether to run the KVM instance in a chroot directory.

If it is set to "true", an empty directory is created before starting the instance and its path is passed via the -chroot flag to kvm. The directory is removed when the instance is stopped.

It is set to "false" by default.

Valid for the KVM hypervisor.

A boolean option that specifies if the instance should be started with its clock set to the localtime of the machine (when true) or to the UTC (When false). The default is false, which is useful for Linux/Unix machines; for Windows OSes, it is recommended to enable this parameter.

Valid for the Xen HVM and KVM hypervisors.

Specifies the address that the VNC listener for this instance should bind to. Valid values are IPv4 addresses. Use the address 0.0.0.0 to bind to all available interfaces (this is the default) or specify the address of one of the interfaces on the node to restrict listening to that interface.

Valid for the Xen HVM and KVM hypervisors.

A boolean option that dictates if the VNC connection is secured with TLS.

Valid for the KVM hypervisor.
